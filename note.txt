Refactoring Plan: Domain-Based CodeMelody with Student/Admin CRUD and E2E Flows
This plan details a major structural refactoring of CodeMelody. Drawing architectural inspiration from modern learning platforms like Coursera and Udemy, we will move from a flat file structure to a clean Domain-Based Architecture. We will introduce a three-tiered user system (Admin, Lecturer, Student), implement complete CRUD operations, and build end-to-end student enrollment pipelines.

Architecture Design: Domain-Based Structure
Rather than having all files located flatly in the root directory, we will partition the application into self-contained domains. This layout prevents cross-domain pollution and ensures the codebase scales cleanly.


codemelody/
│
├── config/
│   ├── config.php                 # Database configuration parameters
│   └── bootstrap.php              # Global bootstrap (sessions, auth checks, helpers)
│

│   └── footer.php                 # Shared layouts footer, modals, and toast scripts

1. Database Schema Extensions
To enable students to log in and support full enrollment flows, we will expand codemelody_database.sql:

Role Extensions: Modify the role enum in the users table to support 'student', 'lecturer', and 'admin'.
Student Linkage: Add a nullable user_id column to the students table referencing users(id). This links student profile details directly to their login credentials.
Enrollments Table: Create a joint table enrollments to track student course registrations:
sql

CREATE TABLE enrollments (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id INT NOT NULL,
    course_id INT NOT NULL,
    enrolled_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    grade VARCHAR(5) DEFAULT NULL,
    FOREIGN KEY (student_id) REFERENCES students(id) ON DELETE CASCADE,
    FOREIGN KEY (course_id) REFERENCES courses(id) ON DELETE CASCADE,
    UNIQUE KEY unique_enrollment (student_id, course_id)
);

**B. Database Tables**
From [codemelody_database.sql](C:/xampp6/htdocs/codemelody/codemelody_database.sql):

`users`
- Stores login identity.
- Columns: `id`, `email`, `password`, `full_name`, `role`, `avatar_initials`, `created_at`.
- Roles: `student`, `lecturer`, `admin`.

`students`
- Stores student profile.
- Columns: `id`, `user_id`, `matric_no`, `full_name`, `email`, `program`, `level`, `enrolled_at`.
- Linked to `users.id`.

`courses`
- Stores course metadata and basic learning content.
- Columns include:
  - `code`, `title`, `credit_units`
  - `lecture_hours`, `practical_hours`
  - `status`, `color_class`
  - `is_paid`, `price`
  - `description`
  - `video_url`
  - `learning_content`
  - `lecturer_id`

`enrollments`
- Join table between students and courses.
- Columns: `id`, `student_id`, `course_id`, `enrolled_at`, `grade`.
- Has unique `student_id + course_id`.

`activities`
- Stores recent activity feed text.
- Columns: `id`, `user_id`, `action_type`, `description`, `created_at`.

**E. Potential Problems**
1. `dashboard.js` references removed legacy modal elements.
   - `openModal()`, `closeModal()`, `createCourse()`, and direct `document.getElementById('createModal').addEventListener(...)` assume `createModal` exists.
   - In the new [templates/footer.php](C:/xampp6/htdocs/codemelody/templates/footer.php), no modal exists.
   - This can cause JavaScript errors on pages.

2. `dashboard.js` still posts to legacy `courses.php`.
   - Root [courses.php](C:/xampp6/htdocs/codemelody/courses.php) is now only a redirect.
   - AJAX course creation/export logic is stale.

3. Sanitization strips HTML from `learning_content`.
   - Good for safety, but limits rich formatted lessons.
   - Current display uses `white-space: pre-line`, not rich text.

4. Course video URL is trusted into an iframe.
   - It is escaped, but not validated against allowed providers.
   - Could lead to bad embeds or unwanted external content.

5. Paid courses are not actually protected by payment.
   - Students can enroll regardless of `is_paid`.

6. Delete operations use GET links.
   - Example: lecturer/student/course delete actions.
   - Should be POST with CSRF protection.

7. No CSRF protection.
   - Forms can be submitted cross-site.

8. No centralized flash messaging.
   - Errors and success messages are per-page or JS-only.

9. Role authorization is present but coarse.
   - More granular policies would help: course owner, enrolled student, admin override.

10. Legacy root pages are half-integrated.
   - Some root feature pages still use old layout files.
   - New domain routes use `templates/`.
   - This creates two UI systems.

11. Database migrations are destructive in the SQL file.
   - It drops all tables.
   - Fine for fresh install, risky for live project updates.

12. Search and notification UI are placeholders.
   - They look real but are not backed by data.
